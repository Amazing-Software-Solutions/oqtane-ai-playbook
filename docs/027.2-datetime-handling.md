# 027.2 - DateTime Handling

Date and time values are **high-risk canonical data**.

Incorrect handling introduces silent data corruption, inconsistent behaviour across users, and defects that are difficult to detect and reproduce.

All DateTime values must be handled in a **context-independent and canonical form** when persisted.

---

## The Core Rule

> **Persist time in UTC.
> Never persist local or context-derived time.**

A DateTime value that depends on timezone, user preference, or environment is not canonical and must not be stored.

---

## Canonical Time

Canonical time is:

- expressed in UTC
- independent of user or system timezone
- consistent across all environments
- safe for persistence and comparison

All persisted DateTime values MUST be canonical.

---

## Contextual Time (Prohibited for Persistence)

Contextual time includes:

- local time (`ToLocalTime()`)
- browser or client-derived time
- user timezone-adjusted values
- server environment timezone values

These representations are valid for display, but never for storage.

---

## Persistence Requirements

All DateTime values written to storage MUST:

- be in UTC
- represent the same moment in time for all users
- be independent of request or execution context
- remain stable regardless of where the code executes

---

## Boundary Violations (Reject on Sight)

The following patterns must be rejected immediately:

- Converting to local time before persistence
- Persisting `DateTime.Now` instead of UTC
- Using server-local timezone values
- Storing values derived from user timezone preferences
- Applying offsets prior to storage

---

## Examples

### Invalid - Local Time Persistence

```csharp
entity.CreatedOn = DateTime.Now;
```

---

### Invalid - Contextual Conversion

```csharp
entity.CreatedOn = utcDate.ToLocalTime();
```

---

### Valid - Canonical UTC Persistence

```csharp
entity.CreatedOn = DateTime.UtcNow;
```

---

### Valid - Preserving Existing UTC Value

```csharp
entity.CreatedOn = utcDate;
```

---

## Display and Transformation

Time transformations are a **presentation concern**.

Timezone conversion and formatting MUST occur:

- on the client
- or at the presentation boundary

Never within persistence logic.

---

## Relationship to Other Rules

- Extends **027.1 Data Integrity Boundaries**
- Operates within **017 Client / Server Responsibility Boundaries**

This rule defines how time is stored, not how it is displayed.

---

## Summary

- Always store time in UTC
- Never persist local or context-derived time
- Perform timezone conversion only for display

DateTime is not just a data type.

It is a consistency boundary.
